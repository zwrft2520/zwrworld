数据库三范式（Normal Forms）是关系型数据库设计中的一组规则，旨在减少数据冗余，提高数据完整性，并改善数据更新的效率。它们是逐步递进的，每一范式都建立在前一范式的基础上。

### 1. 第一范式 (1NF)

**定义：** 数据库表的每一列都是不可再分的原子项。

**解释：** 这意味着表中的每个字段都应该只包含一个值，不能有重复的组或复杂的结构。例如，一个“电话号码”列不应该包含“家庭电话，工作电话”这样的多个电话号码，而应该为每个电话号码单独设置列或单独的行。

**示例：**
**不符合 1NF 的表：**

| 订单ID | 客户姓名 | 商品名称 | 数量 |
| :----- | :------- | :------- | :--- |
| 1      | 张三     | 电视, 冰箱 | 1, 1 |

**符合 1NF 的表：**

| 订单ID | 客户姓名 | 商品名称 | 数量 |
| :----- | :------- | :------- | :--- |
| 1      | 张三     | 电视     | 1    |
| 1      | 张三     | 冰箱     | 1    |

### 2. 第二范式 (2NF)

**定义：** 在满足第一范式的基础上，非主键列必须完全依赖于整个主键，而不是主键的一部分。

**解释：** 2NF 主要针对具有**复合主键**的表。如果一个表的主键由多个列组成，那么表中任何非主键列都不能只依赖于主键中的某一部分，而必须依赖于主键的全部列。如果存在非主键列只依赖于主键的一部分，就说明存在部分函数依赖。

**示例：** 假设有一个学生选课表，主键是 `(学生ID, 课程ID)`。
**不符合 2NF 的表：**

| 学生ID | 课程ID | 学生姓名 | 课程名称 | 教师姓名 |
| :----- | :----- | :------- | :------- | :------- |
| S1     | C1     | 小明     | 数据库   | 李老师   |
| S1     | C2     | 小明     | 编程    | 王老师   |
| S2     | C1     | 小红     | 数据库   | 李老师   |

*   `学生姓名` 只依赖于 `学生ID` (主键的一部分)。
*   `课程名称` 和 `教师姓名` 只依赖于 `课程ID` (主键的一部分)。

**符合 2NF 的表（拆分为多个表）：**

**学生表：**

| 学生ID | 学生姓名 |
| :----- | :------- |
| S1     | 小明     |
| S2     | 小红     |

**课程表：**

| 课程ID | 课程名称 | 教师姓名 |
| :----- | :------- | :------- |
| C1     | 数据库   | 李老师   |
| C2     | 编程     | 王老师   |

**选课表：**

| 学生ID | 课程ID |
| :----- | :----- |
| S1     | C1     |
| S1     | C2     |
| S2     | C1     |

### 3. 第三范式 (3NF)

**定义：** 在满足第二范式的基础上，非主键列之间不能存在传递函数依赖。

**解释：** 传递函数依赖是指，如果存在 A -> B 和 B -> C，那么就存在 A -> C 的传递依赖。在 3NF 中，任何非主键列都不能依赖于其他非主键列，而只能直接依赖于主键。简单来说，就是非主键列之间不能有决定关系。

**示例：** 假设有一个订单详情表，主键是 `订单ID`。
**不符合 3NF 的表：**

| 订单ID | 客户ID | 客户姓名 | 客户地址 |
| :----- | :----- | :------- | :------- |
| 101    | C001   | 张三     | 北京     |
| 102    | C002   | 李四     | 上海     |
| 103    | C001   | 张三     | 北京     |

*   `订单ID` -> `客户ID` (主键决定客户ID)
*   `客户ID` -> `客户姓名` 和 `客户地址` (非主键决定非主键)
*   因此，`订单ID` -> `客户姓名` 和 `订单ID` -> `客户地址` 存在传递依赖。

**符合 3NF 的表（拆分为多个表）：**

**订单表：**

| 订单ID | 客户ID |
| :----- | :----- |
| 101    | C001   |
| 102    | C002   |
| 103    | C001   |

**客户表：**

| 客户ID | 客户姓名 | 客户地址 |
| :----- | :------- | :------- |
| C001   | 张三     | 北京     |
| C002   | 李四     | 上海     |

通过遵守这些范式，可以有效地组织数据库结构，避免数据冗余和更新异常，提高数据库的效率和维护性。
